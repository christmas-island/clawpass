# Releasing clawpass

This project uses [release-plz](https://release-plz.ieni.dev/) for automated
versioning and releases, driven by
[conventional commits](https://www.conventionalcommits.org/).

## How it works

The release pipeline has two stages with clear ownership:

| Stage | Workflow | Trigger | Responsibility |
|-------|----------|---------|----------------|
| **Versioning** | `release-plz.yml` | Push to `main` | Version bump, CHANGELOG, git tag |
| **Artifacts** | `release.yml` | Tag push (`v*`) | Cross-compile binaries, create GitHub Release |

### Flow

1. **Write conventional commits** on feature branches.
2. **Merge to `main`** — `release-plz.yml` runs automatically.
3. **release-plz opens a release PR** that bumps the version in `Cargo.toml`
   and updates `CHANGELOG.md` (via git-cliff).
4. **Merge the release PR** — release-plz creates a git tag (`v<version>`).
5. **Tag push triggers `release.yml`** — builds cross-platform binaries and
   creates the GitHub Release with artifacts attached.

## Conventional commits and version bumps

Commit prefixes determine the version bump:

| Prefix | Bump | Example |
|--------|------|---------|
| `fix:` | Patch | `fix: handle empty queue gracefully` |
| `feat:` | Minor | `feat: add JSON output format` |
| `feat!:` or `BREAKING CHANGE:` | Major | `feat!: change default database path` |
| `docs:`, `ci:`, `chore:`, `test:`, `refactor:` | None (no release) | `docs: update README` |

## Pre-1.0 semantics

While clawpass is below version 1.0.0, the public API is not considered stable.
Breaking changes may land as minor version bumps (`0.x.0`) rather than major
bumps. Pin to a specific version if you depend on clawpass in automation.

## Changelog generation

Changelogs are generated by [git-cliff](https://git-cliff.org/) using the
configuration in `cliff.toml`. Commits are grouped by type (Features, Bug Fixes,
Documentation, etc.) and linked to their GitHub commits.

## Manual override

To force a specific version bump, edit `Cargo.toml` directly in the release PR
before merging. release-plz will respect the version you set.

To trigger a release manually:

```sh
# Install release-plz locally
cargo install release-plz

# Dry run — see what would change
release-plz update --dry-run

# Create a release PR locally (requires GITHUB_TOKEN)
release-plz release-pr
```
